关系型数据库/非关系型数据库

非关系型数据库

  • 特点

    • 易扩展(数据之间无关系)，种类繁多，一个共同的特点都是去掉关系数据库的关系型特性
    • 大数据量，高性能(得益于无关系性/数据结构简单)
    • 灵活的数据模型(NoSQL无需实现为要存储的数据建立字段/随时可以存储自定义的数据格式)
    • 高可用(在不太影响性能的情况下，可以方便的实现高可用的架构[例如：HBase模型、Cassandra，通过复制模型也能实现高可用]

  • 特性

    • key-value`存储系统的跨平台非关系型数据库No SQL

分类

键值(Key-Value)存储

  • 哈希表： 表中有一个特定的键和一个指针指向特定的数据，Key/Value模型简单、已部署(只对部分值查询/更新时，效率低[例如：Redis、Voldmort、Oracle BDB])

列存储数据库

  • 分布式存储的海量数据(键仍然存在，特点是指向了多个列。[如：HBase、Cassandra、Riak])

文档型数据库

  • 数据模型是版本化的文档，半结构化的文档以特定的格式存储(例如：JSON)
  • 可以看作是键值数据的升级版，允许之间嵌套键值，处理网页等复杂数据时，文档型数据库比传统键值数据库的查询效率更高(例如：MongoDB、CouchDB、国内开源文档型数据库SequoiaDB)

图形(Graph)数据库

  • 图形结构的数据库同其他行列的以及刚性结构的SQL数据库不同，它是使用灵活的图形模型，并且能够拓展到多个服务器上
  • NoSQL数据库没有标准的查询语言(SQL)，因此进行数据库查询需要制定数据模型
  • NoSQL数据库部分具有REST式的数据接口或查询API(例如：Neo4J、InfoGrid、Infinite Graph)

适用场景

  1. 数据模型比较简单
  2. 需要灵活性更强的IT系统
  3. 对数据库性能要求较高
  4. 不需要高度的数据一致性(丢失更新、未确定的相关性、不一致的分析和幻读)
  5. 对于给定key，比较容易映射复杂值的环境

关系型数据库

  • 关系型数据库,是指采用了关系模型(简单理解为二维表格模型,一个关系型数据库就是由二维表及其之间的关系组成的一个数据组织)来组织数据的数据库,其以行和列的形式存储数据,以便于用户理解,关系型数据库这一系列的行和列被称为表,一组表组成了数据库。

特点

  1. 存储方式: 表格存储,数据以行和列的方式进行存储;
  2. 存储结构: 结构化的方法存储数据(按照定义好的表结构存入数据,可靠性、稳定性较高),但存入数据后需要修改数据表的结构会十分困难;
  3. 存储规范: 关系型数据库为了避免重复、规范化数据以及充分利用存储空间,单表时按最小关系表的形式存储数据,多表时由于数据表之间存在复杂的关系(数据表数量增加,管理难度随之增加);
  4. 扩展方式: 纵向扩展(由于关系型数据存储在数据表中,数据操作的瓶颈出现在多张数据表的操作中,数据表越多则问题越严重,只能提高处理能力[选择更高性能的计算机,扩展空间有限]);
  5. 查询方式: 结构化查询语言(SQL),采用类似索引(单独的、物理的对数据库表中一列或多列的值进行排序的一种存储结构)加快查询操作;
  6. 规范化: 操作多个数据实体时,规范化数据后再存入到对应的多张关系型数据表中
  7. 事务性:
    1. Atomicity(原子性): 一个操作或是全部完成，或是全部不完成，不会结束在中间某个环节。
    2. Consistency(一致性): 数据库要一直处于一致的状态,事务的运行不会改变数据库原本的一致性约束。
    3. Isolation(隔离性): 事务的执行不被其他事务干扰,操作独立于其他并发操作完成。
    4. Durability(持久性): 操作处理结束后，对数据的修改将永久有效，即使出现系统故障，该修改也不会丢失。
  8. 读写性能: 由于关系型数据库强调数据的一致性,并为此降低读写性能付出了巨大的代价,虽然存储数据和处理数据的可靠性很不错,但当处理海量数据时效率会变得很差,特别是遇到高并发读写时性能下降幅度很高;
  9. 授权方式: Oracle、SQL Server、DB2等都需要支付高昂的费用,即使是免费的MySQL性能也受到了诸多的限制

  • Redis教程: https://www.runoob.com/redis/redis-tutorial.html

Redis特性(相比其他key-value缓存产品有以下三个特点)

  • Redis支持数据的持久化,可以将内存中的数据保存在磁盘中,重启的时候可以再次加载进行使用;
  • Redis不仅仅支持简单的key-value类型的数据,同时还提供list/set/zset/hash等数据结构的存储;
  • Redis支持数据的备份,即master-slave模式的数据备份。

事务一致性

  • 强一致性: 系统中的某个数据被成功更新后,后续的访问都能看到更新后的值;
  • 弱一致性: 系统中的某个数据被更新后,后续的访问可能得到更新后的值,也可能是更改前的值;
  • 最终一致性: 系统中的某个数据被更新,经过一段时间后,最终所有的访问都是更新的值.

ACID

  1. Atomicity(原子性): 一个操作或是全部完成，或是全部不完成，不会结束在中间某个环节。
  2. Consistency(一致性): 数据库要一直处于一致的状态,事务的运行不会改变数据库原本的一致性约束。
  3. Isolation(隔离性): 事务的执行不被其他事务干扰,操作独立于其他并发操作完成。
  4. Durability(持久性): 操作处理结束后，对数据的修改将永久有效，即使出现系统故障，该修改也不会丢失。

CAP/BASE 定理

参考:https://developer.aliyun.com/article/692238

CAP

  • 一个分布式系统中,一致性(Consistency)、可用性(Availability)、分区容错性(Partition tolerance),这三个要素最多只能同时实现两点,不可能三者兼顾

一致性(Consistency)

  • 定义: 在分布式系统中的所有数据备份,在同一时刻是否同样的值。(等同于所有节点访问同一份最新的数据副本)

  • 说明: 分布式系统中,数据一般会存在不同节点的副本中。如果对第一个节点的数据成功进行了更新操作,而第二个节点上的数据却没有得到相应的更新,这时候读取第二个节点的数据依然是更新前的数据,即脏数据,这就是分布式系统数据不一致的情况。
    在分布式系统中,如果能够做到针对一个数据项的更新操作执行成功后,所有的用户都能读取到最新的值,那么这样的系统就被认为具有强一致性(或严格一致性)

可用性(Availability)

  • 定义: 保证每个请求不管成功或者失败都有响应

分区容错性(Partition tolerance)

  • 定义: 系统中任意信息的丢失或失败不会影响系统的继续运作
  • 说明: 数据存在的节点越多,分区容忍性越高,但要复制更新的数据就越多,一致性就越难保证。为了保证一致性,更新所有节点数据所需要的时间就越长,可用性就会降低。

BASE

  • BASE理论是对CAP理论的延伸,思想是即使无法做到强一致性(CAP的一致性就是强一致性),但可以采用适当的采取弱一致性,即最终一致性。
  • BASE是指基本可用(Basically Available)、软状态(Soft State)、最终一致性(Eventual Consistency)

基本可用

  • 基本可用是指分布式系统在出现故障的时候,允许损失部分可用性(例如响应时间、功能上的可用性),允许损失部分可用性。基本可用绝不等价于系统不可用

软状态

  • 软状态是指允许系统存在中间状态,而该中间状态不会影响系统整体可用性。分布式存储中一般一份数据会有多个副本,允许不同副本同步的延时就是软状态的体现。

最终一致性

  • 最终一直性是指系统中的所有数据副本经过一段时间后,最终能够达到一致的状态。弱一致性和强一致性相反,最终一致性是弱一致性的一种特殊情况
