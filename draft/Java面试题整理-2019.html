Java篇

定义章节

  1. Java语言的特点
    1. 简单易学;
    2. 面向对象(封装、继承、多台);
    3. 平台无关性(Java虚拟机实现与平台无关);
    4. 可靠性;
    5. 安全性;
    6. 支持多线程(C++ 语言没有内置的多线程机制,因此必须调用操作系统的多线程功能来进行多线程程序设计,而Java语言却提供了多线程支持);
    7. 支持网络编程并且很方便(Java语言诞生本身就是为简化网络编程设计的,因此Java语言不仅支持网络编程而且很方便);
    8. 编译与解释并存。

  2. JVM、JDK、JRE

    JVM

    Java虚拟机(JVM)是运行Java字节码的虚拟机。JVM有针对不同系统的特定实现(Windows、Linux、macOS),目的是使用相同的字节码,都能给出相同的结果

    JDK

    JDK是Java Development Kit,它是功能齐全的Java SDK。它拥有JRE所拥有的一切,还有编译器(javac)和工具(如:javadoc和jdb)。它能够创建和编译程序。

    JRE

    JRE是Java运行时环境。它是运行已编译Java程序所需的所有内容的集合,包括Java虚拟机(JVM)、Java类库、Java命令和其他的一些基础构建。但是,它不能用于创建新程序。

  3. Oracle JDK和Open JDK的对比

    1. Oracle JDK版本将每三年发布一次,而Open JDK版本每三个月发布一次;
    2. Open JDK是一个参考模型并且是完全开源的,而Oracle JDK是Open JDK的一个实现,并不是完全开源的;
    3. Oracle JDK比Open JDK更稳定,同时代码非常接近。在安全性上Oracle JDK有着更多的类和一些错误修饰,如果进行企业/商业软件开发建议选择Oracle JDK;
    4. Oracle JDK不仅响应性比Open JDK强,还提供了更优越的JVM性能;
    5. Oracle JDK不会为即将发布的版本提供长期支持,需要通过更新获取更高版本;
    6. Oracle JDK根据二进制代码许可协议获得许可,而Open JDK根据GPL v2获得许可。

  • 形式上: 字符常量是单引号标注的一个字符;字符串是双引号标注的若干字符(字符常量:'a';字符串常量:"abcd")

  1. 构造器Constructor是否可被override?
    不能被override(重写),可以被overload(重载)
  2. 重载和重写
  • 重载: 同一个类中,方法名相同、参数类型、个数、顺序不同,方法返回值和访问修饰符可以不同
  • 重写: 发生在父子类中;实现未用abstract修饰的interface类,方法名、参数名列表必须相同,返回值范围小于等于父类,抛出的异常范围小于等于父类,方位修饰符大于等于父类;如果父类方法访问修饰符为private则子类就不能重写该方法。
  1. String、StringBuffer、StringBuilder
  • 可变性: String类使用final修饰字符串数组保存字符串,不可变;StringBuffer、StringBuilder继承自AbstractStringBuilder类,都没有用finale修改,都可变
  • 线程安全: String线程安全;StringBuffer对方法加了同步锁或者对调用的方法加了同步锁,也是线程安全的;StringBuilder并没有对方法加同步锁,非线程安全
  • 性能: String类型进行改变时,每次都会生成新的String对象同时指针指向新的String对象;StringBuffer每次都会对StringBuffer对象本身操作,而不是生成新的对象并改变对象引用。相同情况下使用StringBuilder相比StringBuilder仅能获得10%~15%的性能提升,需承担多线程不安全风险

总结:

操作少量数据: String
单线程操作字符串缓冲区下操作大量数据: StringBuilder
多线程操作字符串缓冲区下操作大量数据: StringBuffer

  14. 转发(Forward)和重定向(Redirect)的区别
    1. 地址栏显示不同:
      • forward: 服务器请求资源,由服务器直接访问目标地址的URL,把URL响应内容发送给浏览器,浏览器不知道响应来源; 地址栏不变。
      • redirect: 服务器端根据逻辑,发送一个状态码,告诉浏览器重新去请求那个地址;地址显示的是新的URL
    2. 数据共享
      • forward: 转发页面和转发到的页面可以共享request里面的数据
      • redirect: 不能共享数据
    3. 运用的地方
      • forward: 一般用于用户登陆的时候,根据角色转发到响应的模式
      • redirect: 一般用于用户注销登录时返回主页面和跳转到其他网站等
    4. 效率
      • forward: 高
      • redirect: 低

  7. HashMap和Hashtable的区别

    1. 线程是否安全: HashMap是非线程安全的,HashTable是线程安全的;HashTable内部的方法基本都经过synchronized修饰

  1. 线程、程序、进程
  2. 线程的生命周期和装填
    1. NEW: 初始状态,线程被构建,但是还没有调用start()方法
    2. RUNNABLE: 运行状态,Java线程将操作系统中的就绪和运行两种状态笼统地称作运行中
    3. BLOCKED: 阻塞状态,表示线程阻塞于锁
    4. WAITING: 等待状态,表示线程进入等待状态,进入该状态表示当前线程需要等待其他线程作出一些特定动作(通知或中断)
    5. TIME_WAITING: 超时等待状态,该状态不同于WAITING,它是可以在指定的时间自行返回的
    6. TERMINATED: 终止状态,表示当前线程已经执行完毕
  3. 并行和并发的区别
    • 并发: 同一时间段内,多个任务都在执行(单位时间内不一定同时执行)
    • 并行: 单位时间内,多个任务同时执行
  4. 为什么使用多线程
    1. 计算机底层: 线程可以比作是轻量级的进程,是程序执行的最小单位,线程间的切换和调度的成本远远小于进程。另外,多核CPU时代意味着多个线程可以同时运行,减少了线程上下文切换的开销
    2. 互联网发展趋势: 系统要求的并发量高,多线程并发编程是开发高并发系统的基础,利用好多线程机制可以大大提高系统整体的并发能力以及性能
  5. 多线程带来的问题: 并发编程的目的是为了能提高程序的执行效率和提高程序运行速度,但是并发编程并不总是能提高程序运行速度的;可能会遇到很多问题,比如: 内存泄漏、上下文切换、死锁还有受限于硬件和软件的资源闲置问题
  6. 为什么调用start()方法会执行run()方法,为什么不能直接调用run()方法
    • 调用start方法方可启动线程并使线程进入就绪状态,而run()方法只是thread的一个普通方法调用,还是在主线程里执行。

高并发与多线程

高并发与多线程的关系与区别

  1. 什么是高并发

    • 高并发(High Concurrency) 是一种系统运行过程中遇到的一种短时间内遇到大量操作请求的情况,主要发生在web系统集中大量访问收到大量请求(例如：12306的抢票、天猫双十一活动)。该情况发生会导致系统在这段时间内执行大量操作,例如对资源的请求、数据库的操作等。

  2. 高并发处理指标

    • 响应时间(Response Time):系统请求做出响应的时间。例如系统处理一个HTTP请求需要200 ms,200 ms即为系统的响应时间
    • 吞吐量(Throughput): 单位时间内处理的请求数量
    • 每秒查询率QPS(Query Per Second): 每秒响应请求数。在互联网领域,这个指标和吞吐量区分的没有这么明显。
    • 并发用户数: 同时承载正常使用系统功能的用户数量。例如一个即时通讯系统,同时在线量一定程度上代表了系统的并发用户数。

  3. 高并发 、多线程的关系和区别

    • 多线程: 多线程是处理高并发的一种编程方法,即并发需要多线程实现
    • 高并发: 高并发并不是Java专有，是与语言无关的广义的，为提供更好互联网服务而提出的概念。
    • 多线程对应的是CPU,高并发对应的是访问请求

ACID

  • 原子性(Atomicity): 一个事务要么全部执行,要么不执行。
  • 一致性(Consistency): 事务的运行并不改变数据库中的一致性。
  • 独立性(Isolation): 事务的独立性也有称作隔离性,是指两个以上的事务不会出现交错执行的状态。因为这样可能会导致数据不一致。
  • 持久性(Durability): 事务的持久性是指事务执行成功以后,该事务对数据库所作的更改便是持久的保存在数据库之中。

问题

  1. 从浏览器地址栏输入http://www.baidu.com后回车,到我们看到搜索页面,中间经历了什么?
    1. 经历了域名解析
    2. TCP^1三次握手
      1. 客户端发送SYN(SEQ = x)报文给服务器端,进入SYN_SEND^2状态,等待服务器确认;
      2. 服务器端接收到SYN报文,必须确认客户的SYN( ack = x +1 ),同时自己也发送一个SYN包(syn = y),即SYN + ACK包,此时服务器进入SYN_RECV^3状态;
      3. 客户端收到服务器端的SYN + ACK包,向服务器发送确认包ACK(ACK = y + 1),此包发送完毕,客户端和服务器进入Established^4状态,完成三次握手。
    3. 建立TCP 连接后发起HTTP请求
    4. 服务器响应HTTP请求
    5. 浏览器解析HTML代码,同时请求HTML代码中的资源(如:js、css、图片等)
    6. 浏览器对页面进行渲染呈现给用户

  • QPS(Queries Per Second): 每秒响应请求数

默认情况下建立TCP连接不会断开,只有在请求报头中声明Connection:close才会在接受完成后关闭连接

  1. 一个TCP连接可以对应几个请求?

如果维持连接,一个TCP可以发送多个HTTP请求

  1. 一个TCP连接中HTTP请求发送可以一起发送么(比如一起发送三个请求,再三个响应一起接收)?

在HTTp/1.1存在Pipelining技术可以完成这个多个请求同时发送,但是由于浏览器默认关闭,所以可以认为这是不可行的。在HTTP2中由于Multiplexing特点的存在,多个HTTP请求可以在同一个TCP连接中并行连接

  1. 为什么有的时候刷新页面不需要重新建立SSL连接?

TCP连接有的时候会被浏览器和服务端维持一段时间,TCP不再需要重新建立,SSL自然也会用之前的。

  1. 浏览器对同一Host建立TCP连接的数量有没有限制?

有。Chrome最多允许对同一个Host建立六个TCP连接,不同的浏览器有一些区别

消息队列

初级问题

  1. 定义
    • 消息队列是一种能帮助开发人员解决系统间异步通信的中间件,常用于解决系统解耦和请求的削峰平谷的问题。它是面向研发人员的一种中间件,虽然不能直接创造价值,但可以有效的简化开发工作。
  2. 为什么使用消息队列?
    1. 通过异步处理提高系统性能(削峰、减少响应所需时间),
      1. 将短时间内高并发产生的事务消息存储在消息队列中,从而削平高峰期的并发事务;
    2. 降低系统耦合性
  3. 消息队列的弊端
    • 系统可用性降低: 系统可用性在某种程度上降低,在加入MQ之前,不用考虑消息丢失或MQ挂掉等情况,但是引入MQ之后就需要考虑这些问题
    • 系统复杂性提高: 加入MQ之后,需要保证消息没有被重复消费、处理消息丢失的情况、保证消息传递的顺序性等问题
    • 一致性问题: 消息队列通过异步实现,虽然可以提高系统响应速度,但是无法保证消息的真正消费者正确消费消息。将会导致数据不一致,产生一致性问题。
